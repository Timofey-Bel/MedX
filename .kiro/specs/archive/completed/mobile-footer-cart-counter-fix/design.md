# Mobile Footer Cart Counter Fix - Bugfix Design

## Overview

На странице товара в мобильной версии сайта счетчик корзины в футере не обновляется при добавлении товара в корзину. Проблема заключается в том, что Knockout.js bindings не применяются к элементу `.mobile-cart-counter` в мобильном футере, в то время как десктопный счетчик `.cart-counter` в header работает корректно.

Приложение использует глобальную модель `CartCounterViewModel` (Knockout.js) для управления счетчиками корзины. При инициализации скрипт `cart-counter.js` ищет элементы с классами `.cart-counter` и `.mobile-cart-counter` и применяет к ним Knockout bindings. Однако мобильный футер использует другой класс или структуру, из-за чего bindings не применяются.

Стратегия исправления: добавить правильный класс к счетчику корзины в мобильном футере и убедиться, что Knockout bindings применяются корректно.

## Glossary

- **Bug_Condition (C)**: Условие, при котором проявляется баг - когда пользователь добавляет товар в корзину на странице товара в мобильной версии, счетчик в футере не обновляется
- **Property (P)**: Желаемое поведение - счетчик корзины в мобильном футере должен обновляться синхронно с изменениями корзины
- **Preservation**: Существующее поведение десктопного счетчика и всей функциональности корзины должно остаться неизменным
- **CartCounterViewModel**: Глобальная Knockout.js модель в `legacy/assets/sfera/js/cart-counter.js`, управляющая отображением счетчиков корзины
- **mobile-cart-counter**: CSS класс, который должен быть применен к счетчику корзины в мобильном футере для применения Knockout bindings
- **data-bind**: Атрибут Knockout.js для связывания DOM элементов с ViewModel свойствами

## Bug Details

### Fault Condition

Баг проявляется, когда пользователь добавляет товар в корзину на странице товара в мобильной версии сайта. Счетчик корзины в мобильном футере не обновляется, потому что к нему не применены Knockout.js bindings.

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type CartUpdateEvent
  OUTPUT: boolean
  
  RETURN input.deviceType == 'mobile'
         AND input.page == 'product'
         AND input.action IN ['add_to_cart', 'update_quantity', 'remove_from_cart']
         AND mobileFooterCounterExists()
         AND NOT knockoutBindingsApplied('.mobile-cart-counter')
END FUNCTION
```

### Examples

- **Пример 1**: Пользователь на мобильном устройстве открывает страницу товара, нажимает "В корзину" → счетчик в футере остается "0" или не отображается, хотя товар добавлен в корзину
- **Пример 2**: Пользователь на мобильном устройстве увеличивает количество товара с 1 до 3 → счетчик в футере не обновляется с "1" на "3"
- **Пример 3**: Пользователь на мобильном устройстве удаляет товар из корзины → счетчик в футере не обновляется и продолжает показывать старое значение
- **Пример 4 (Edge case)**: Пользователь обновляет страницу после добавления товара → счетчик в футере показывает корректное значение (загруженное с сервера при инициализации)

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- Десктопный счетчик корзины в header должен продолжать работать точно так же
- Основная функциональность корзины (добавление, обновление, удаление товаров) должна остаться неизменной
- Все другие счетчики корзины на сайте (если есть) должны продолжать работать корректно
- Визуальное отображение мобильного футера должно остаться неизменным
- Инициализация CartCounterViewModel и загрузка данных с сервера должны работать как раньше

**Scope:**
Все взаимодействия с корзиной, которые НЕ происходят на странице товара в мобильной версии, должны быть полностью не затронуты этим исправлением. Это включает:
- Добавление товаров в корзину на десктопной версии
- Работу со страницей корзины (cart.blade.php)
- Обновление счетчиков на других страницах (каталог, главная, избранное)
- Работу других компонентов мобильного футера (навигация, избранное, профиль)

## Hypothesized Root Cause

На основе анализа кода, наиболее вероятные причины бага:

1. **Отсутствие класса mobile-cart-counter**: В файле `resources/views/components/mobile-bottom-nav.blade.php` счетчик корзины может не иметь класса `.mobile-cart-counter`, который ищет скрипт `cart-counter.js` при инициализации
   - Скрипт применяет bindings к элементам с селектором `.cart-counter, .mobile-cart-counter`
   - Если элемент не имеет этого класса, bindings не применяются

2. **Отсутствие data-bind атрибутов**: Элемент счетчика может не иметь необходимых Knockout.js атрибутов `data-bind="text: formattedCount, visible: isVisible"`
   - Без этих атрибутов Knockout не знает, какие свойства ViewModel связывать с элементом

3. **Порядок загрузки скриптов**: Скрипт `cart-counter.js` может загружаться после рендеринга мобильного футера, но до того, как элемент появится в DOM
   - Это маловероятно, так как скрипт использует `DOMContentLoaded` event

4. **Конфликт с другими скриптами**: Другой скрипт может перезаписывать содержимое счетчика или удалять bindings
   - Это маловероятно, так как десктопный счетчик работает корректно

## Correctness Properties

Property 1: Fault Condition - Mobile Footer Cart Counter Updates

_For any_ cart update event (add, update quantity, remove) that occurs on the product page in mobile view, the fixed mobile footer cart counter SHALL immediately update to reflect the new cart item count, displaying the correct total number of items in the cart.

**Validates: Requirements 2.1, 2.2**

Property 2: Preservation - Desktop and Other Functionality

_For any_ cart update event that does NOT occur on the product page in mobile view (desktop cart updates, other page cart updates, other mobile footer components), the fixed code SHALL produce exactly the same behavior as the original code, preserving all existing cart counter functionality on desktop, all cart operations, and all other mobile footer navigation items.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5**

## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct (отсутствие класса и data-bind атрибутов):

**File**: `resources/views/components/mobile-bottom-nav.blade.php`

**Element**: Счетчик корзины в мобильном футере (внутри ссылки на корзину)

**Specific Changes**:
1. **Добавить класс mobile-cart-counter**: Убедиться, что элемент `<span>` счетчика имеет класс `mobile-cart-counter`
   - Текущий код: `<span class="mobile-nav-badge mobile-cart-counter" data-bind="text: formattedCount, visible: isVisible"></span>`
   - Проверить, что класс присутствует и написан правильно

2. **Проверить data-bind атрибуты**: Убедиться, что элемент имеет корректные Knockout.js bindings
   - Должен быть: `data-bind="text: formattedCount, visible: isVisible"`
   - Эти атрибуты связывают элемент с свойствами CartCounterViewModel

3. **Проверить структуру HTML**: Убедиться, что элемент счетчика находится в правильном месте в DOM
   - Должен быть внутри ссылки на корзину
   - Не должен быть скрыт другими стилями или элементами

4. **Проверить загрузку скрипта cart-counter.js**: Убедиться, что скрипт загружается на странице товара
   - Проверить, что скрипт включен в layout или в секцию scripts страницы товара
   - Проверить порядок загрузки (должен быть после Knockout.js)

5. **Добавить отладочные логи (временно)**: Для подтверждения, что bindings применяются
   - Раскомментировать console.log в cart-counter.js для отладки
   - После исправления - закомментировать обратно

## Testing Strategy

### Validation Approach

Стратегия тестирования следует двухфазному подходу: сначала воспроизвести баг на неисправленном коде и понять причину, затем проверить, что исправление работает корректно и не нарушает существующую функциональность.

### Exploratory Fault Condition Checking

**Goal**: Воспроизвести баг ПЕРЕД внесением исправлений. Подтвердить или опровергнуть анализ первопричины. Если опровергнем, нужно будет пересмотреть гипотезу.

**Test Plan**: Открыть страницу товара в мобильной версии (используя DevTools или реальное устройство), проверить HTML структуру счетчика корзины, добавить товар в корзину и наблюдать, обновляется ли счетчик. Проверить в консоли браузера, применены ли Knockout bindings к элементу.

**Test Cases**:
1. **HTML Structure Test**: Открыть страницу товара в мобильной версии, проверить в DevTools, что элемент счетчика имеет класс `mobile-cart-counter` и атрибут `data-bind` (может не иметь на неисправленном коде)
2. **Knockout Bindings Test**: В консоли браузера выполнить `ko.dataFor(document.querySelector('.mobile-cart-counter'))` - должен вернуть `undefined` на неисправленном коде, если bindings не применены
3. **Add to Cart Test**: Добавить товар в корзину на странице товара в мобильной версии - счетчик не обновится (будет показывать старое значение или быть скрытым)
4. **Page Refresh Test**: Обновить страницу после добавления товара - счетчик покажет корректное значение (загруженное с сервера)

**Expected Counterexamples**:
- Элемент счетчика не имеет класса `mobile-cart-counter` или атрибута `data-bind`
- Knockout bindings не применены к элементу (ko.dataFor возвращает undefined)
- Счетчик не обновляется при добавлении товара в корзину
- Возможные причины: отсутствие класса, отсутствие data-bind атрибутов, неправильный селектор в cart-counter.js

### Fix Checking

**Goal**: Проверить, что для всех входных данных, где выполняется условие бага, исправленная функция производит ожидаемое поведение.

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := mobileFooterCartCounter_fixed(input)
  ASSERT counterUpdatesImmediately(result)
  ASSERT counterShowsCorrectValue(result)
END FOR
```

**Test Plan**: После внесения исправлений, повторить тесты на странице товара в мобильной версии, проверить, что счетчик обновляется при всех операциях с корзиной.

**Test Cases**:
1. **Add Single Item**: Добавить 1 товар в корзину → счетчик должен показать "1"
2. **Add Multiple Items**: Добавить 3 единицы товара → счетчик должен показать "3"
3. **Update Quantity**: Увеличить количество с 1 до 5 → счетчик должен обновиться с "1" на "5"
4. **Remove Item**: Удалить товар из корзины → счетчик должен обновиться (уменьшиться или скрыться)
5. **Multiple Products**: Добавить несколько разных товаров → счетчик должен показывать общее количество всех товаров

### Preservation Checking

**Goal**: Проверить, что для всех входных данных, где условие бага НЕ выполняется, исправленная функция производит тот же результат, что и оригинальная функция.

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT originalBehavior(input) = fixedBehavior(input)
END FOR
```

**Testing Approach**: Property-based testing рекомендуется для проверки сохранения поведения, потому что:
- Автоматически генерирует множество тестовых случаев по всему домену входных данных
- Находит граничные случаи, которые могут быть упущены в ручных unit тестах
- Предоставляет сильные гарантии, что поведение не изменилось для всех не-багованных входных данных

**Test Plan**: Наблюдать поведение на НЕИСПРАВЛЕННОМ коде для десктопных счетчиков и других взаимодействий, затем написать property-based тесты, фиксирующие это поведение.

**Test Cases**:
1. **Desktop Counter Preservation**: Проверить, что десктопный счетчик в header продолжает работать корректно на всех страницах (главная, каталог, товар, корзина)
2. **Other Mobile Footer Items Preservation**: Проверить, что другие элементы мобильного футера (Главная, Каталог, Избранное, Профиль) работают как раньше
3. **Cart Page Preservation**: Проверить, что страница корзины работает корректно (обновление количества, удаление товаров, применение промокодов)
4. **Favorites Counter Preservation**: Проверить, что счетчик избранного в мобильном футере продолжает работать корректно

### Unit Tests

- Тест проверки наличия класса `mobile-cart-counter` в HTML
- Тест проверки наличия атрибута `data-bind` с корректными значениями
- Тест применения Knockout bindings к элементу счетчика
- Тест обновления счетчика при добавлении товара (mock CartCounterViewModel)
- Тест обновления счетчика при изменении количества товара
- Тест обновления счетчика при удалении товара

### Property-Based Tests

- Генерировать случайные операции с корзиной (add, update, remove) и проверять, что счетчик всегда показывает корректное значение
- Генерировать случайные комбинации товаров и количеств, проверять, что счетчик показывает правильную сумму
- Тестировать на разных размерах экрана (mobile, tablet, desktop) и проверять, что соответствующие счетчики обновляются

### Integration Tests

- Полный флоу: открыть страницу товара в мобильной версии → добавить товар → проверить счетчик в футере → перейти в корзину → проверить содержимое
- Тест переключения между десктопной и мобильной версией: проверить, что счетчики синхронизированы
- Тест обновления страницы: добавить товар → обновить страницу → проверить, что счетчик показывает корректное значение
- Тест множественных операций: добавить товар → увеличить количество → удалить → добавить снова → проверить счетчик на каждом шаге
