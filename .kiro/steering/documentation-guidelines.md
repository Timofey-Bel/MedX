# Руководство по документированию модулей

## Обязательное требование

**КРИТИЧЕСКИ ВАЖНО:** При создании или изменении любого модуля, контроллера, сервиса или раздела приложения **ОБЯЗАТЕЛЬНО** создавать документацию в папке `docs/`.

## Структура документации

### Расположение файлов

Все документы создаются в папке `c:\OS\home\sfera\docs\` (относительный путь: `docs/`).

### Именование файлов

Используйте kebab-case для имен файлов:
- `module-name.md` - для модулей
- `controller-name.md` - для контроллеров
- `service-name.md` - для сервисов
- `feature-name.md` - для функциональности

**Примеры:**
- `docs/favorites-module.md`
- `docs/cart-module.md`
- `docs/auth-controller.md`
- `docs/payment-service.md`

## Когда создавать документацию

### Обязательно документировать:

1. **Новые модули** - при создании нового функционального модуля
2. **Новые контроллеры** - при создании контроллера с несколькими методами
3. **Новые сервисы** - при создании сервиса с бизнес-логикой
4. **API endpoints** - при создании новых API endpoints
5. **Сложная логика** - при реализации сложной бизнес-логики
6. **Интеграции** - при интеграции с внешними сервисами

### Можно не документировать:

1. Простые helper функции
2. Миграции БД (они самодокументируемые)
3. Конфигурационные файлы
4. Простые модели без логики

## Что включать в документацию

### Обязательные разделы:

1. **Обзор** - что делает модуль
2. **Архитектура** - структура кода
3. **API Endpoints** - если есть API
4. **База данных** - если работает с БД
5. **Требования** - ссылки на требования из спецификации

### Опциональные разделы:

1. **Тестирование** - если есть тесты
2. **Известные ограничения** - если есть
3. **Будущие улучшения** - если запланированы
4. **Примеры использования** - если сложный API

## Стиль документации

### Язык

- Документация на **русском языке**
- Технические термины можно оставлять на английском
- Примеры кода с комментариями на русском

### Форматирование

- Используйте Markdown
- Заголовки: `#` для главных, `##` для разделов, `###` для подразделов
- Код: используйте блоки кода с указанием языка
- Списки: используйте `-` для ненумерованных, `1.` для нумерованных
- Таблицы: используйте Markdown таблицы для структурированных данных

### Примеры кода

Всегда включайте примеры:
- Запросов к API
- Ответов от API
- SQL запросов
- Использования методов

```php
// Пример использования FavoriteService
$favoriteService = app(FavoriteService::class);
$result = $favoriteService->addItem($productId, $userId);

if ($result['success']) {
    echo "Товар добавлен в избранное";
}
```

## Обновление документации

### Когда обновлять:

1. При добавлении новых методов
2. При изменении API endpoints
3. При изменении структуры БД
4. При исправлении багов, влияющих на поведение
5. При добавлении новых зависимостей

### Как обновлять:

1. Добавьте запись в раздел "История изменений"
2. Обновите соответствующие разделы
3. Обновите дату последнего изменения
4. Если изменения критичные - добавьте предупреждение в начало документа

## Проверка документации

Перед завершением задачи убедитесь:

- ✅ Документация создана в папке `docs/`
- ✅ Все обязательные разделы заполнены
- ✅ Примеры кода работают
- ✅ API endpoints документированы с примерами
- ✅ Ссылки на требования из спецификации указаны
- ✅ История изменений обновлена

## Важные замечания

1. **Документация - это код** - она должна быть актуальной и точной
2. **Пишите для других** - представьте, что читатель не знаком с проектом
3. **Будьте конкретны** - избегайте общих фраз, приводите примеры
4. **Обновляйте регулярно** - устаревшая документация хуже, чем её отсутствие
5. **Используйте диаграммы** - если логика сложная, добавьте схемы (Mermaid)

## Ответственность

- **Разработчик** - создает и обновляет документацию при изменении кода
- **Reviewer** - проверяет наличие и качество документации при code review
- **Tech Lead** - следит за общей структурой и полнотой документации
