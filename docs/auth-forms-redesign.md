# Переделка форм регистрации и логина

**Дата**: 2026-03-05  
**Задача**: Переделать формы регистрации и логина, используя дизайн из legacy шаблонов с добавлением функционала регистрации организаций

## Проблема

Текущие формы регистрации и логина имели простой дизайн без преимуществ и социальных кнопок. Пользователь указал, что legacy формы (`legacy/site/modules/sfera/register/register.tpl` и `legacy/site/modules/sfera/login/login.tpl`) выглядят симпатичнее.

Требования:
1. Использовать двухколоночный layout из legacy (форма слева, преимущества справа)
2. Добавить социальные кнопки входа (Google, VK, Telegram, Сбер ID)
3. Добавить чекбокс "Регистрация организации" вместо переключателя типа пользователя
4. При включении чекбокса показывать поля ИНН и данных организации
5. Интеграция с существующим `DaDataService` для автозаполнения данных по ИНН

### 3. JavaScript функционал

#### `public/assets/sfera/js/auth-register-new.js`

Новый файл для работы с чекбоксом регистрации организации:

**Функционал:**
- Переключение видимости полей организации при изменении чекбокса
- Изменение `user_type` (retail/wholesale) в зависимости от чекбокса
- Установка/снятие обязательности полей организации
- Интеграция с `DaDataInnChecker` для проверки ИНН
- Автозаполнение полей организации из DaData
- Блокировка полей после успешной проверки ИНН
- Кнопка "Изменить ИНН" для сброса проверки
- Валидация ИНН (только цифры, макс 12 символов)
- Показ/скрытие пароля
- Toast уведомления

**Логика работы:**
1. По умолчанию чекбокс выключен, `user_type = 'retail'`
2. При включении чекбокса:
   - Показываются поля организации (`.organization-fields.visible`)
   - `user_type` меняется на `'wholesale'`
   - Поля ИНН и полного наименования становятся обязательными
3. При выключении чекбокса:
   - Скрываются поля организации
   - `user_type` возвращается к `'retail'`
   - Очищаются все поля организации
   - Снимается обязательность

### 4. Интеграция с существующим функционалом

**Используемые сервисы:**
- `DaDataService` - проверка ИНН и получение данных организации
- `DaDataInnChecker` (JS) - клиентская часть проверки ИНН

**Используемые скрипты:**
- `auth-register.js` - базовый функционал регистрации (индикатор силы пароля)
- `dadata-inn-check.js` - модуль проверки ИНН через DaData API
- `auth-register-new.js` - новый функционал с чекбоксом организации

**Контроллер:**
- `AuthController::store()` - уже поддерживает регистрацию организаций
- Проверяет `user_type` и создает запись в `orgs` если `wholesale`

## Тестирование

### Сценарий 1: Регистрация розничного покупателя

1. Открыть `/register`
2. Заполнить: имя, телефон, email, пароль
3. Чекбокс "Регистрация организации" выключен
4. Нажать "Зарегистрироваться"
5. Ожидаемый результат: создан пользователь с `user_type = 'retail'`, `org_id = NULL`

### Сценарий 2: Регистрация оптового покупателя

1. Открыть `/register`
2. Заполнить: имя, телефон, email, пароль
3. Включить чекбокс "Регистрация организации"
4. Появляются поля организации
5. Ввести ИНН: `7707083893` (Яндекс)
6. Нажать "Проверить ИНН"
7. Поля автоматически заполняются данными из DaData
8. Нажать "Зарегистрироваться"
9. Ожидаемый результат:
   - Создана запись в `orgs` с данными организации
   - Создан пользователь с `user_type = 'wholesale'`, `org_id = <id организации>`
   - Перенаправление в `/lk/` (оптовый кабинет)

### Сценарий 3: Вход в систему

1. Открыть `/login`
2. Ввести email и пароль
3. Нажать "Войти"
4. Ожидаемый результат:
   - Розничный покупатель → `/my/` (розничный кабинет)
   - Оптовый покупатель → `/lk/` (оптовый кабинет)
